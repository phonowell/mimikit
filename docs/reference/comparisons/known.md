# 项目对比分析（Mimikit / OpenClaw / Nanobot）

## 对比结论（面向当前决策）

| 维度 | Mimikit | OpenClaw | Nanobot |
| --- | --- | --- | --- |
| 工程定位 | 单 session 轻量助手 | 大生态平台型助手 | 超轻量实验型助手 |
| 架构复杂度 | 中（manager/worker） | 高（多模块多扩展） | 低（微内核） |
| 可控性 | 高（私有闭环 + 约束明确） | 中（生态强但治理成本高） | 中高（体量小，边界清晰） |
| 扩展能力 | 中（当前以 CLI/WebUI 为主） | 高（平台/技能生态广） | 中（可扩展但生态弱） |

## 对 Mimikit 的高 ROI 启示

1. 优先补“长期记忆检索能力”，而不是先扩展渠道。
2. 先做部署成本降低（脚本化/容器化），再做生态扩张。
3. 保持单 session 主架构，避免提前引入多路由系统复杂度。
4. provider 路由按“稳定性优先”渐进增强，避免一次性全量改造。

## 优先级

- `P0`：长期记忆检索、部署简化。
- `P1`：多 provider 路由与安全边界显式化。
- `P2`：外部平台接入与可视化增强。
